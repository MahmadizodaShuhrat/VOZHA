# 🔧 ТЗ: Grace period при разрыве WebSocket админа комнаты

## Проблема

Сейчас, когда у создателя комнаты (админа) разрывается WebSocket-соединение — например, при переключении приложения в фон (что происходит каждый раз при открытии share-меню для отправки пригласительной ссылки в Telegram/WhatsApp) — бэкенд **моментально удаляет комнату**.

В результате:

1. Пользователь A создаёт комнату, получает код `616593`
2. Пользователь A нажимает **"Share"** → открывается share-меню → приложение уходит в фон
3. MIUI / Xiaomi / Android battery saver приостанавливает сетевую активность фонового приложения
4. Сокет пользователя A разрывается
5. Бэкенд видит разрыв → считает что "админ вышел" → **удаляет комнату**
6. Пользователь A возвращается в приложение — переподключается, но комнаты уже нет
7. Пользователь B переходит по ссылке → "Комната не найдена"

## Воспроизведение (подтверждено на реальных устройствах)

| Шаг | Телефон 1 (создатель) | Телефон 2 (приглашённый) |
|-----|----------------------|--------------------------|
| 1 | Создаёт комнату, код = 616593 | — |
| 2 | Нажимает Share → Telegram | — |
| 3 | Приложение ушло в фон — **разрыв сокета** | — |
| 4 | — | Получил ссылку в Telegram, нажал на неё |
| 5 | — | Приложение открылось, код автозаполнился, но сервер ответил **"room_not_found"** |

Обходной путь (работает сейчас): не переключать приложение в фон; код комнаты диктовать вручную второму игроку. Это плохой UX.

---

## Требуемое поведение

### Grace period на разрыве админа

Когда WebSocket админа разрывается — **не удалять комнату сразу**. Вместо этого:

1. Пометить сессию как "отключённую" (admin_disconnected_at = now)
2. Запустить таймер на **60 секунд**
3. Если за эти 60 секунд создатель **переподключился** (новый WS-сокет от того же user_id) → восстановить сессию, отменить удаление, продолжить работу комнаты как ни в чём не бывало
4. Если 60 секунд прошло без реконнекта → только тогда удалить комнату и отправить `delete_room` оставшимся участникам

### Поведение в деталях

**При разрыве WS админа**:
```pseudocode
on_admin_socket_close(room):
  if room.game_started:
    # Игра уже идёт — ждём дольше (2 минуты), всё равно участники могут доиграть
    room.admin_disconnected_at = now
    schedule_cleanup(room, delay=120s)
  else:
    # Ещё ожидание в waiting room
    room.admin_disconnected_at = now
    schedule_cleanup(room, delay=60s)
  
  # НЕ отправляем delete_room другим участникам пока
```

**При реконнекте того же user_id**:
```pseudocode
on_ws_connect(user_id, socket):
  existing_room = find_room_where_admin_is(user_id)
  if existing_room and existing_room.admin_disconnected_at:
    # Админ вернулся — восстанавливаем
    existing_room.admin_disconnected_at = null
    cancel_cleanup_timer(existing_room)
    existing_room.admin_socket = socket
    # Отправить ему текущее состояние комнаты (текущий список участников, phase, etc)
    send(socket, 'room_restored', existing_room.state)
```

**Истечение grace period**:
```pseudocode
on_cleanup_timer_fires(room):
  if room.admin_disconnected_at is null:
    # Админ успел вернуться — таймер уже отменён, сюда не попадаем
    return
  # Grace прошёл, админа нет
  broadcast(room.members, 'delete_room')
  remove_from_public_list(room)
  delete_room(room)
```

### Что это должно давать клиенту

- Отправка ссылки в Telegram → приложение в фон → сокет падает → возвращение из фона (3-5 секунд) → сокет восстанавливается → **комната живёт**
- Если создатель действительно вышел и не вернулся за минуту → только тогда комната закрывается
- `delete_room` больше не приходит остальным участникам "сразу" — только после grace period, что соответствует реальному "админ ушёл"

## Клиент (уже подготовлен)

На клиенте уже есть автореконнект (`BattleWsService._tryReconnect`):
- Каждые 10 секунд — ping
- При разрыве — до 5 попыток реконнекта с backoff 1s, 2s, 3s, 4s, 5s
- Максимально клиент восстановится за ~15 секунд после возвращения из фона

Это укладывается в 60-секундное окно grace period, так что дополнительной работы на клиенте **не нужно**, если бэкенд внедрит grace period.

### Опционально — улучшение для клиента

Если бэкенд также пришлёт событие `room_restored` при успешном реконнекте — клиент может показать короткий тост "Соединение восстановлено". Но без этого тоже всё будет работать: на клиенте просто не отобразится ошибка.

## Приоритет

**🔴 Критический** — ломает основной сценарий: приглашение друга через мессенджер. Это главный способ привлечения новых игроков в multiplayer-комнаты.

## ⚠️ Баг-репорт от 2026-04-24 (после внедрения grace period)

После внедрения grace period на бэкенде и матчинг-клиента — клиент уже обрабатывает `room_restored` и шлёт аутентифицированный `check_room` на реконнекте — на реальных устройствах воспроизводится сценарий:

### Шаги воспроизведения
1. **Phone 1** (создатель): создаёт комнату → получает код, например `840650` → waiting room.
2. **Phone 1**: жмёт Share → Telegram → отправляет ссылку `vozhaomuz://battle?room_id=840650` второму телефону.
3. **Phone 2**: открывает Telegram, тапает на ссылку.
4. **Phone 2**: приложение открывается, Battle > Join Room, код автозаполнен, автоматически срабатывает `check_room` → сервер отвечает `check_room` (успех) → UI показывает карточку комнаты с именем создателя и кнопкой "Присоединиться".
5. **Phone 2**: пользователь жмёт "Присоединиться" → клиент шлёт `join_to_room`.
6. **Phone 2 получает `join_new_member`** от сервера — UI показывает waiting room с ДВУМЯ участниками: создатель + сам Phone 2. Это значит `join_to_room` **успешно обработан на сервере** — комната реально содержит обоих.
7. **Phone 1 НЕ получает `join_new_member`**. У создателя в списке участников по-прежнему только он. Из-за того что WS админа в grace-состоянии (conn == nil), `sendMessageToAllRoomMembers` его пропускает — это логично с точки зрения спеки.
8. **≈ 5–6 секунд спустя**: Phone 2 получает событие `room_not_found` (а не `delete_room`!), UI показывает ошибку "Хона ёфт нашуд" (room not found). Phone 2 выкидывается из комнаты.

### Критические наблюдения

- **Вопрос 1**: Почему `room_not_found`, а не `delete_room`? Эти события семантически разные, и клиент их рендерит по-разному. По нашей спеке (и по предоставленному backend описанию) при истечении admin grace должно приходить `delete_room`. Откуда `room_not_found`?
- **Вопрос 2**: 5–6 секунд — это явно не 60-секундный grace. Что за таймаут?
- **Вопрос 3**: Когда Phone 1 всё-таки возвращается из Telegram (допустим, через 10 сек после Share), он должен получить `room_restored` с обновлённым списком (включая Phone 2, который вошёл в grace-окно). Нужно убедиться что **server действительно запоминает новых members, добавленных во время admin grace**, и передаёт их в `room_restored.data.members`.

### Ожидаемое поведение
- Phone 2 остаётся в комнате до истечения admin grace (60 сек).
- Если Phone 1 возвращается до истечения grace — получает `room_restored` с Phone 2 в списке, комната продолжает нормально работать, Phone 2 видит Phone 1 снова онлайн (без разрывов).
- Если grace истекает без возврата Phone 1 — комната удаляется, Phone 2 получает `delete_room` (не `room_not_found`), UI показывает "Создатель закрыл комнату".

### Гипотезы (что проверить в бэкенде)

1. **Grace период действительно ~5 секунд, а не 60**. Проверить, что `ROOM_ADMIN_GRACE_WAITING_SECONDS` (или дефолт `RoomAdminGraceWaiting`) установлен в 60. Может, env-переменная не подхватилась / переопределена в проде.

2. **`join_to_room` отклоняется во время grace** — сервер считает комнату "в процессе удаления" и возвращает `room_not_found`. Это вроде не описано в нашей спеке, но возможно реализовано по-другому. **Нужно**: во время grace `join_to_room` должен работать штатно — просто новый член добавляется, `join_new_member` рассылается всем с `conn != nil` (Phone 1 конечно не получит пока админ в grace, но запомнит в state комнаты).

3. **`tryRestoreAdminSession` срабатывает повторно** — возможно, когда Phone 2 шлёт `check_room` с своим JWT, сервер ошибочно считает это попыткой восстановить админ-сессию и перезапускает таймер / сбрасывает что-то. Проверить: tryRestoreAdminSession должна срабатывать только если `userID == room.CreatorID`. Любой другой userID на том же сокете (или на новом сокете в той же комнате) — обычный `check_room` / `join_to_room`, без побочных эффектов на admin session.

4. **Race condition между check_room и join_to_room**:
   - T=0: Phone 2 шлёт `check_room` → сервер читает room из памяти, отвечает ок
   - T+1s: Phone 1 вернулся из Telegram, сокет реконнектится, шлёт свой `check_room` (от клиента)
   - T+2s: `tryRestoreAdminSession` → восстановление сессии
   - T+3s: Phone 2 жмёт Join → шлёт `join_to_room`
   - T+4s: Что-то идёт не так — возможно grace уже expired из-за неправильного таймера или race в `finalizeAdminLeft`

5. **Логирование**: добавить лог на сервере при каждом:
   - `onAdminSocketClose` (startAdminDisconnectedAt, expected expiry time)
   - `tryRestoreAdminSession` (cancel timer, new socket)
   - `finalizeAdminLeft` (grace expired, broadcasting delete_room)
   - `join_to_room` (during grace или после)
   - `check_room` (during grace)
   - При `room_not_found` — **обязательно** логировать *почему* (room deleted, never existed, admin in grace too long, etc).

### Что нужно от бэкенд-команды

- **Воспроизвести** сценарий 1–7 выше в staging (если есть доступ) или локально с двух устройств
- **Добавить логирование** по пункту 5 выше
- **Проверить** env-переменные `ROOM_ADMIN_GRACE_WAITING_SECONDS` на проде — должно быть ≥ 60
- **Ответить**: должен ли `join_to_room` работать во время admin grace? Если нет — тогда сценарий "Share → Friend joins" будет всегда падать, пока админ в фоне. Это тогда вообще не решаемо со стороны клиента (share процесс занимает хотя бы 3-5 сек минимум).

### Клиент-сторона — что уже сделано

- ✅ Обрабатывает `room_restored` и ресинхронизирует state
- ✅ На реконнекте WS посылает `check_room` с JWT (чтобы `tryRestoreAdminSession` сработал)
- ✅ Реконнект работает в фазах `waitingRoom`, `countdown`, `playing`, `waitingResults`, `reconnecting`

Дополнительно на стороне клиента, если backend подтвердит что это их баг:

- Добавить snackbar "Соединение восстановлено" при получении `room_restored`
- Показать более осмысленное сообщение при `room_not_found` во время join-потока: "Создатель комнаты отключился, попробуйте ещё раз" (или что-то подобное, если это действительно причина)
