# 📋 Главное ТЗ для Backend: все известные проблемы VozhaOmuz

Этот документ — **сводный список всех задач** для бэкенд-команды. Каждая задача ссылается на отдельный детальный ТЗ в папке `docs/`, а здесь даётся краткая суть и приоритет.

Клиентская сторона по каждой задаче либо уже готова, либо есть понятный план обновления — сигналы ниже.

---

## 🔴 КРИТИЧЕСКИЕ — ломают основной сценарий

### 1. Grace period при разрыве WS админа

**Статус**: ⚠️ Backend внедрил grace логику, клиент обновлён — но на реальном тесте воспроизводится баг: при `Phone 1 share → Phone 2 join via deeplink`, Phone 2 получает `room_not_found` через 5-6 секунд, а Phone 1 не видит, что кто-то заходил.

**Баг-репорт**: раздел "⚠️ Баг-репорт" в [TZ_BACKEND_ROOM_DISCONNECT_GRACE.md](TZ_BACKEND_ROOM_DISCONNECT_GRACE.md). В нём — 5 гипотез, что проверить, включая env-переменные, race conditions, логирование, и ключевой вопрос: **должен ли `join_to_room` работать во время admin grace**.

**Детали**: [TZ_BACKEND_ROOM_DISCONNECT_GRACE.md](TZ_BACKEND_ROOM_DISCONNECT_GRACE.md)

### 2. Deep link — публикация верификационных файлов на api.vozhaomuz.com

**Проблема**: Сейчас `https://api.vozhaomuz.com/api/v1/deeplink?page=battle&room_id=X` возвращает JSON. Когда пользователь нажимает на ссылку в Telegram/WhatsApp — приложение не открывается, браузер открывает голый JSON.

Мы сделали временный фолбэк на GitHub Pages (`https://mahmadizodashuhrat.github.io/vozhaomuz-landing/`), чтобы пользователи уже могли делиться ссылками. Но для продакшна правильно — на собственном домене.

**Что нужно сделать** (на `api.vozhaomuz.com`):

| URL | Содержимое | Content-Type |
|-----|------------|--------------|
| `/.well-known/assetlinks.json` | JSON с SHA-256 release + debug | `application/json` |
| `/.well-known/apple-app-site-association` | JSON с Apple Team ID + bundle ID (**без** расширения `.json` в URL!) | `application/json` |
| `/api/v1/deeplink?page=X&room_id=Y` | HTML landing-страница (не JSON) с JS-редиректом на `vozhaomuz://` | `text/html` |

Все файлы готовы в папке [`deeplink-landing/`](../deeplink-landing/) — можно забрать как есть. Внутри уже:
- `assetlinks.json` с настоящими production + debug SHA-256
- `apple-app-site-association` шаблон (замените `REPLACE_WITH_TEAM_ID`)
- `index.html` — готовая landing с пульсирующей попыткой открыть custom scheme + фолбэком на Play Store / App Store

**Детали**: [TZ_DEEP_LINK_BACKEND.md](TZ_DEEP_LINK_BACKEND.md)

**После публикации**: клиент переключим обратно на `api.vozhaomuz.com` — AndroidManifest + iOS entitlements + share text (3 файла, 10 минут).

### 3. Правильный расчёт результатов и призов

**Проблема**: таблица лидеров сортируется только по `score`, ties разрешаются случайно (Dart `List.sort` нестабилен). Нет полей `place` / `finish_time` в payload `results`. Приз выплачивается как приходит — нет прозрачной формулы. Длительность матча ~90 секунд, а по ТЗ должно быть 4 минуты.

**Что нужно сделать**:

| Изменение | Что дать клиенту |
|-----------|------------------|
| Длительность матча = 240 сек | `end_time - start_time == 240` в `start_game` |
| Трекинг `finish_time` по игроку | Новое поле в `MemberDto`, ISO-8601 UTC |
| Детерминированная сортировка | `has_left ↑` → `score ↓` → `correct ↓` → `answered ↓` → `finish_time ↑` → `id ↑` |
| Поле `place` в `results` | 1, 2, 3, … для реальных доигравших. `null` для ботов и ушедших |
| Приз 50 / 30 / 20 | `pool = money_count × count(real_humans)`, распределение по `place` |
| Edge cases | 2 доигравших → 70/30; 1 → 100%; 0 → сгорает; 1 человек + боты → 100% pool |

**Клиент**:
- Новое поле `int? place` и `DateTime? finishTime` в `MemberDto` — добавим при релизе.
- Сортировку на клиенте заменим на чтение `place` с сервера. Если поле null (старый бэкенд) — fallback на текущую сортировку по score.
- Время прохождения теста (`finish_time - start_time`) добавим в колонку таблицы лидеров + подиум + summary.

**Детали** (со всеми формулами, edge case-ами, 8 тест-кейсами): [TZ_BACKEND_BATTLE_RANKING_AND_PRIZES.md](TZ_BACKEND_BATTLE_RANKING_AND_PRIZES.md)

## 🟠 ВЫСОКИЕ — ломают отдельные функции

### 4. Длительность ожидания в waiting room — серверное поле

**Проблема**: Клиент раньше показывал 60-секундный таймер "жду игроков", хардкод. Мы убрали визуальный таймер, оставили только watchdog 180 сек как safety net. Теперь там просто "Ожидание игроков…" без конкретного числа.

**Что нужно**: в событии `room_created` прислать `wait_time_seconds` — сколько секунд сервер будет ждать перед добавлением ботов / удалением комнаты. Клиент покажет это число в UI countdown'ом.

```json
{
  "type": "room_created",
  "data": {
    "room_id": "534825",
    "wait_time_seconds": 60,   // ← НОВОЕ (int, сколько ждём)
    ...
  }
}
```

**Клиент**: добавим визуальный таймер обратно, читающий `wait_time_seconds`. Если поле null — оставим "Ожидание игроков…" без числа.

**Приоритет**: Средний, но после фикса бекенд-таймера (сейчас ~90 сек, должно быть 60 для public-rooms) желательно сразу и поле добавить — чтобы клиент показал точное число.

### 5. Structured `daily_limit_reached` ответ

**Проблема**: Сейчас при превышении дневного лимита (3 попытки battle) сервер шлёт текстовое поле `message` на русском. Это попадает в UI у тадж/английских пользователей как кириллица.

**Что нужно**: отдельный тип события или стабильный код ответа, не локализованный текст:

```json
{
  "type": "daily_limit_reached",
  "data": {
    "limit": 3,           // сколько раз в день разрешено
    "resets_at": "2026-04-25T00:00:00Z"  // когда сбросится
  }
}
```

**Клиент**: уже перехватывает type=`daily_limit_reached` и подменяет на свой локализованный текст — работает. Дополнительные поля (`limit`, `resets_at`) позволят показать точнее: "Вы использовали 3/3 попытки. Сброс через 4ч 12м".

### 6. Поле `days_active` в `/profile-rating`

**Проблема**: Разные варианты endpoint-а возвращают разные поля — иногда `daysActive`, иногда `days_active`, иногда `active_days`, иногда `streak`. Это — **дневной streak** (последовательные дни активности), а не общее число сыгранных дней.

**Что нужно**: зафиксировать одно имя поля. Рекомендуется `current_streak` или `days_active`, причём значение — **последовательные дни активности** (то, что показывается огоньком на home screen).

**Клиент**: сейчас читает из `/user/activity` вместо `profile-rating`, потому что там поле стабильное. Но было бы правильно чтобы оба endpoint'а согласовались — и streak можно было брать из любого.

## 🟡 СРЕДНИЕ — можно отложить

### 7. Атомарное обновление money в syncProgress

**Проблема**: После завершения учебной сессии клиент делает оптимистичное обновление money (добавляет заработанные монеты локально), потом дожидается `getProfile()` и синхронизируется с сервером. Если `getProfile()` вернёт старые данные до того, как backend обработал начисление — optimistic update затрётся.

**Что нужно**: в ответе `syncProgress` присылать `new_balance` — точное значение денег после начисления:

```json
{
  "count": 15,                 // сколько монет начислено
  "new_balance": 1845,         // ← НОВОЕ, итоговый баланс после начисления
  "new_achievements": [...]
}
```

**Клиент**: используем `new_balance` напрямую, нет никаких optimistic-race-condition-ов.

### 8. Конфигурируемый battle daily limit

**Проблема**: Сейчас лимит в 3 попытки/день для non-premium зашит на клиенте (`_freeDailyJoins = 2`). Продукту может захотеться изменить (например, для новых пользователей в первый день — 10 попыток).

**Что нужно**: отдавать лимит в `/profile-rating` или `/user/settings`:

```json
{
  "battle_daily_limit": 2,    // сколько попыток в день у non-premium
  "battle_daily_remaining": 1  // сколько осталось сегодня
}
```

**Клиент**: будем читать оттуда вместо хардкода. `remaining` позволит показать "Осталось 1 битва на сегодня".

## 🟢 НИЗКИЕ — когда будет время

### 9. `count_learned_words` в profile-rating — формула одинакова с клиентом

**Проблема**: Иногда расходится: сервер считает все слова с `state > 0`, клиент считает с `state == 2`. Результат: счётчики на разных устройствах разные.

**Что нужно**: зафиксировать единую формулу. Предлагаю считать "выучено" = `state >= 1` (и клиенту сделать то же самое). Согласовать с продуктом.

### 10. Version description — локализованные поля

**Проблема**: Сейчас `/version-info` возвращает `description` в одном языке. Нужен lookup по локали.

**Что нужно**:

```json
{
  "version": "1.4.2",
  "update_required": false,
  "description": {
    "tg": "Навигоркарди нав…",
    "ru": "Обновления…",
    "en": "Updates…"
  }
}
```

**Клиент**: уже готов. Берёт `description[locale]` с фолбэком на `tg` → любой доступный.
