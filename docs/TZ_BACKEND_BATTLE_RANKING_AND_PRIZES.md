# 🏆 ТЗ: Правильный расчёт ранжирования и призов в Battle

## Контекст

В режиме Battle несколько игроков играют одновременно и в конце видят таблицу лидеров и награду. Сейчас есть несколько проблем:

1. **Место в таблице определяется только по score** — нет tie-breaker'ов. Если два игрока набрали одинаково, порядок в топ-3 получается случайным (Dart `List.sort` нестабильный).
2. **`won_coins` приходит с бэкенда, но алгоритм непрозрачный** — клиент не знает, по какой логике распределяется приз. Нет гарантии, что при одинаковых score суммарный фонд не выплатится дважды.
3. **Нет поля `finish_time`** — сервер не присылает, когда игрок закончил тест. Без этого невозможно различить двух игроков с одинаковым score: быстрее ответивший должен быть выше.
4. **Длительность матча зашита где-то непрозрачно** — клиент просто берёт `start_time` / `end_time` из `start_game`. Нужно явно зафиксировать 4 минуты.

Этот документ описывает требования к бэкенду, чтобы таблица результатов и призы были **детерминированными, справедливыми и проверяемыми**.

## Требования

### 1. Длительность матча — 4 минуты

При старте игры (`start_game`):

```
start_time = now (server UTC)
end_time   = start_time + 4 minutes  // ровно 240 секунд
```

Клиент уже корректно парсит `start_time` / `end_time` из ответа (с `.toUtc()`). Никаких изменений на клиенте не требуется — только на сервере зафиксировать 240 секунд вместо текущих ~90 секунд.

### 2. Трекинг `finish_time` для каждого участника

Сервер обязан записывать момент, когда игрок закончил тест:

- **Триггер записи**: либо игрок отправил последний (N-й) ответ через `answered`, либо прислал `finish_test` вручную, либо истёк `end_time` матча (в этом случае `finish_time = end_time` для всех, кто не успел закончить).
- **Источник времени**: серверные часы UTC. Клиентское время не использовать — нет доверия, легко подделывается.
- **Формат**: ISO-8601 UTC, миллисекундная точность (`2026-04-24T10:03:42.531Z`).

### 3. Детерминированное ранжирование

Все игроки сортируются по следующему chain-у tie-breaker'ов в строгом порядке:

```
ORDER BY
  has_left      ASC,   -- 1) ушедшие в самом низу
  score         DESC,  -- 2) больше очков → выше
  correct_answers DESC, -- 3) при равенстве — больше правильных
  answered      DESC,  -- 4) при равенстве — больше отвеченных вопросов
  finish_time   ASC,   -- 5) при равенстве — кто быстрее закончил
  id            ASC    -- 6) последний fallback для стабильности
```

**Важно**: `has_left` идёт первым — ушедшие всегда в конце таблицы, независимо от того, сколько очков они успели набрать. Они не имеют права на приз.

### 4. Поле `place` в каждом `MemberDto`

Сервер обязан сам вычислять место и присылать его во всех событиях, которые несут финальную статистику матча:
- `results`
- `finish_test` (если клиент явно отправил `finish_test`)

Новые поля на каждом `MemberDto` в payload:

```json
{
  "id": 123,
  "name": "Azamat",
  "score": 180,
  "correct_answers": 18,
  "answered": 20,
  "won_coins": 2400,
  "has_left": false,
  "place": 1,                              // ← НОВОЕ
  "finish_time": "2026-04-24T10:03:42.531Z" // ← НОВОЕ
}
```

Правила назначения `place`:

- **Боты** (`id < 0`) → `place = null`. Боты **не участвуют в нумерации мест** — они отображаются отдельно как "бот-соперники" без номера.
- **Ушедшие** (`has_left: true`) → `place = null`. Они в конце таблицы, без места.
- **Реальные активные игроки** (id > 0, `has_left: false`) → `place = 1, 2, 3, …` в порядке из п. 3.

То есть `place` — это позиция **среди людей, доигравших матч**. Именно эту позицию клиент использует для подиума, медалей и призов.

### 5. Распределение приза — 50 / 30 / 20

**Призовой фонд** (pool):
```
pool = money_count × count(real_human_members)

где real_human_members = игроки с id > 0 (включая тех, кто has_left)
```

Обоснование: только люди платят entry fee. Боты не платят — они в фонд не вносят. Ушедшие свой взнос теряют: он остаётся в фонде для призёров.

**Распределение** (при 3+ доигравших людях):

| `place` | Доля | Пример (pool = 1000) |
|---------|------|----------------------|
| 1       | 50%  | 500 монет |
| 2       | 30%  | 300 монет |
| 3       | 20%  | 200 монет |
| 4+ / null | 0  | 0 монет |

Все игроки с `place >= 4` или `place == null` получают `won_coins = 0`.

**Округление**: дробные части (при делении нечётных сумм) округляются **в пользу игрока — вверх**, с защитой чтобы сумма выплат не превысила pool:

```
first  = round_half_up(pool × 0.50)
second = round_half_up(pool × 0.30)
third  = pool - first - second            // остаток целиком 3-му
```

Это гарантирует, что `first + second + third == pool` (нет потерянных монет, нет переплат).

### 6. Обработка tied-мест (одинаковый ранг)

**Критическая ситуация**: два игрока имеют абсолютно одинаковые score + correct + answered + finish_time. Такое почти невозможно, но возможно (пинг-совпадение).

**Правило**: при полном равенстве tie-breaker fallback на `id ASC` (п. 3) гарантирует, что **всегда ровно один игрок на каждом месте** — никаких дубликатов. Это самое простое, детерминированное и безопасное правило.

Альтернатива — "равный дележ призового пула при ничьей" — усложняет логику и открывает баги. Не используем.

### 7. Edge cases

**7.1. Меньше 3 доигравших людей**

"Доиграл" = `id > 0` **и** `has_left: false` (бот или ушедший — не считается).

| Доигравших людей | Распределение |
|------------------|---------------|
| 0 | Приз никому (pool "сгорает") |
| 1 | 1-е место = 100% pool |
| 2 | 1-е = 70%, 2-е = 30% |
| 3+ | 50 / 30 / 20 (стандарт) |

**7.2. Боты**

Полностью вынесено в п. 4 и п. 5:
- `place = null` (не участвуют в ranking'е людей)
- `won_coins = 0`
- Не вносят в pool (не платят entry fee)

Ботов клиент отображает отдельным блоком ("Bot-opponent"), без номера места и без медали.

Пример: 4 участника — бот, человек А (score=200), человек Б (score=150), человек В (score=100).
- Бот: place=null, won_coins=0
- А: place=1, won_coins = 50% pool
- Б: place=2, won_coins = 30% pool
- В: place=3, won_coins = 20% pool

Если вдруг бот набрал больше А по score — это не меняет расклад: бот всё равно `place=null`, а А остаётся 1-м среди людей.

**7.3. Ушедшие игроки**

`has_left: true` → `place = null`, `won_coins = 0`. Entry fee не возвращается — он уходит в общий pool. Сервер не начисляет им монеты.

Важно: ушедший игрок всё равно приходит в `results.members` (клиенту нужно отобразить его в конце таблицы с бейджем "⨯ вышел"), просто без места и без выплаты.

**7.4. Все игроки ушли**

Если все участники вышли до `end_time`, никто не получает приз. Pool "сгорает" (остаётся у системы). Событие `results` всё равно шлётся — клиенты должны корректно отобразить пустую таблицу.

**7.5. Никто не ответил ни на один вопрос**

Если несколько игроков имеют `score = 0, correct = 0, answered = 0` — ранжирование идёт по `finish_time ASC`. Обычно в этом случае `finish_time = end_time` (истёк таймер), и дальше по `id ASC`. Так или иначе — выплата идёт по порядку, дубликатов нет.

**7.6. Один человек + боты**

Распространённый случай (комната заполнилась ботами после `wait_time_seconds`). Человек — `place=1`, won_coins = 100% pool (потому что 1 доигравший человек — п. 7.1, строка `1 | 100% pool`). Pool = money_count × 1 (только он платил).

## JSON contract — изменения

### Событие `results` (сервер → клиент)

**До** (текущее):
```json
{
  "type": "results",
  "data": {
    "members": [
      { "id": 1, "name": "A", "score": 180, "correct_answers": 18, "answered": 20, "won_coins": 500 }
    ]
  }
}
```

**После**:
```json
{
  "type": "results",
  "data": {
    "members": [
      {
        "id": 1,
        "name": "A",
        "score": 180,
        "correct_answers": 18,
        "answered": 20,
        "won_coins": 500,
        "has_left": false,
        "place": 1,
        "finish_time": "2026-04-24T10:03:42.531Z"
      }
    ]
  }
}
```

Поля `place` и `finish_time` **обязательные** для всех членов в payload:
- `place`: `int` или `null` (для ботов и ушедших)
- `finish_time`: ISO-8601 UTC или `null` (если игрок не закончил — например `has_left: true` до истечения таймера)

### Событие `finish_test` (сервер → клиент)

Если бэкенд использует `finish_test` как отдельное событие со списком участников (отличное от `results`) — те же два поля (`place`, `finish_time`) **обязательны** и там.

### Событие `start_game` (сервер → клиент)

Контракт не меняется. Изменение только в значениях:

```
end_time - start_time == 240 секунд  // ровно 4 минуты
```

## Тест-кейсы (проверить после внедрения)

Обозначения: `money_count = 10` везде (entry fee 10 монет), если не указано иное.

### Кейс 1: Обычный матч, 5 реальных игроков, разные score
- Участники (id > 0, has_left=false): A, B, C, D, E
- Pool = 10 × 5 = 50 монет
- Финальные score / correct / answered / finish_time:
  - A: 250 / 20 / 20 / 10:01:50
  - B: 200 / 18 / 20 / 10:02:15
  - C: 200 / 18 / 20 / 10:02:30  ← равен B по score+correct+answered
  - D: 150 / 15 / 20 / 10:03:00
  - E: 80  / 8  / 15 / 10:04:00
- Ожидание:
  - A: place=1, won_coins=25 (50% от 50)
  - B: place=2, won_coins=15 (30%) — раньше закончил чем C → выше
  - C: place=3, won_coins=10 (20%)
  - D: place=4, won_coins=0
  - E: place=5, won_coins=0
- Сумма выплат: 25+15+10 = 50 ✓

### Кейс 2: Ничья двух игроков на 1-м месте (только двое в комнате)
- Участники: A, B
- Pool = 10 × 2 = 20 монет
- score=200, correct=18, answered=20 — у обоих
- finish_time: A=10:03:00, B=10:03:15
- Ожидание (применяется правило §7.1 "2 доигравших → 70/30"):
  - A: place=1, won_coins = round(20 × 0.70) = 14
  - B: place=2, won_coins = 20 − 14 = 6
- Сумма выплат: 14+6 = 20 ✓

### Кейс 3: Комната с ботами и людьми
- Участники: bot1 (id=-1), humanA (id=5), humanB (id=6), humanC (id=7)
- Pool = 10 × 3 = 30 монет (бот не платит)
- Финальные score: bot1=300, humanA=200, humanB=150, humanC=100
- Ожидание:
  - bot1: place=null, won_coins=0
  - humanA: place=1, won_coins=15 (50% от 30)
  - humanB: place=2, won_coins=9  (30%)
  - humanC: place=3, won_coins=30 − 15 − 9 = 6 (20% с учётом остатка)
- Сумма выплат к людям: 15+9+6 = 30 ✓
- Бот отображается в UI без номера места, без медали

### Кейс 4: Массовый уход
- Участники: 10 реальных людей
- Pool = 10 × 10 = 100 монет
- Ушли 7 человек в процессе, доиграли 3 (A, B, C)
- Ожидание:
  - A: place=1, won_coins=50 (50%)
  - B: place=2, won_coins=30 (30%)
  - C: place=3, won_coins=20 (20%)
  - 7 ушедших: place=null, won_coins=0, присутствуют в таблице с бейджем "⨯ вышел"
- Сумма выплат: 50+30+20 = 100 ✓

### Кейс 5: 200-player матч
- Участники: 200 реальных игроков
- money_count = 10, pool = 10 × 200 = 2000 монет
- Все доиграли
- Ожидание:
  - place=1: won_coins=1000 (50%)
  - place=2: won_coins=600  (30%)
  - place=3: won_coins=400  (20%)
  - place=4..200: won_coins=0
- Сумма выплат: 1000+600+400 = 2000 ✓

### Кейс 6: Один человек + комната из ботов
- Участники: human (id=42), bot1, bot2, bot3 (все id < 0)
- Pool = 10 × 1 = 10 монет (только человек платил)
- Неважно, сколько набрал человек — он единственный "доигравший человек" (§7.1)
- Ожидание:
  - human: place=1, won_coins = 10 (100% pool)
  - bot1, bot2, bot3: place=null, won_coins=0
- Сумма выплат: 10 ✓

### Кейс 7: Grace period — разрыв и реконнект админа
- Админ создал комнату, WS разорвался (ушёл в share-меню)
- Параллельная спецификация: [TZ_BACKEND_ROOM_DISCONNECT_GRACE.md](TZ_BACKEND_ROOM_DISCONNECT_GRACE.md)
- Провести матч до конца **после** реконнекта и убедиться:
  - `place`, `finish_time`, `won_coins` посчитаны корректно
  - Админ не потерял свой результат
  - Для остальных участников таблица не изменилась

### Кейс 8: Округление при нечётном pool
- money_count = 7, 5 игроков → pool = 35
- Ожидание:
  - 1-е: round(35 × 0.5) = round(17.5) = 18 (round half up)
  - 2-е: round(35 × 0.3) = round(10.5) = 11 (round half up)
  - 3-е: 35 − 18 − 11 = 6 (остаток)
- Сумма: 18+11+6 = 35 ✓ (инвариант `sum == pool`)

## Приоритет

**🔴 Высокий** — без детерминированного ранжирования таблица лидеров выглядит случайно, что подрывает доверие. Без явного `place` клиент вынужден вычислять место сам, и если его алгоритм разойдётся с серверным — у игроков на разных устройствах разные "первые места". Это критично для турниров/соревнований.

## Что делает клиент

1. **Сейчас**: клиент сортирует только по `score`, rendering нестабильный. Prize-пул не вычисляется — доверяем `won_coins` от сервера как есть.
2. **После внедрения ТЗ**: клиент использует серверное поле `place` напрямую (без собственной сортировки). Никаких клиентских формул нет — все выплаты и ранжирования авторитетны на сервере.
3. **Если клиент старый** (ещё не обновлён): продолжит сортировать сам по `score`, но `place` от сервера просто проигнорирует. Это не поломает старые APK — backwards-compatible.

Клиентский патч для перехода на серверный `place`:
- Файл: [`lib/feature/battle/data/member_dto.dart`](../lib/feature/battle/data/member_dto.dart) — добавить поле `int? place` и `DateTime? finishTime`.
- Файл: [`lib/feature/battle/presentation/screens/battle_finished_page.dart`](../lib/feature/battle/presentation/screens/battle_finished_page.dart) — заменить ручной `sort((a, b) => b.score.compareTo(a.score))` на:

```dart
sort((a, b) {
  // place == null (боты, ушедшие) → уезжают в конец таблицы
  final ap = a.place ?? 0x7fffffff;
  final bp = b.place ?? 0x7fffffff;
  final byPlace = ap.compareTo(bp);
  if (byPlace != 0) return byPlace;
  // Тай-брейк для стабильности между ботами / ушедшими (у них у всех place=null)
  return a.id.compareTo(b.id);
});
```

Эти правки мы подготовим и выпустим в следующем релизе, как только бэкенд начнёт слать новые поля.

## UI-требования (что сохраняем / что добавляем на экране результатов)

### Сохраняем как есть

Экран результатов (Unity 3D-стиль: подиум + таблица лидеров) **визуально не меняется**. Все существующие анимации остаются:

1. **ShieldPlaceAnimation** — большой щит-медаль с анимацией для места игрока (1-е / 2-е / 3-е / 4+). Рендерится на основе `place` текущего пользователя (если `place == null` — игрок бот или ушёл, анимация не рендерится).
2. **Подиум top-3** — золото / серебро / бронза только для игроков с `place ∈ {1, 2, 3}`. Боты и ушедшие на подиум не попадают.
3. **Таблица лидеров** — полный список всех участников (включая ботов и ушедших). У реальных доигравших отображается номер `place`; у ботов — бейдж "Bot", у ушедших — бейдж "⨯ вышел", без номера.

**Единственное изменение в логике UI**: вместо "место = индекс в отсортированном массиве + 1" (текущее) → "место = `member.place` (из payload сервера)". Если сервер пока не шлёт `place` (старый бэкенд) — клиент fallback-ит на текущую сортировку по score.

Анимации, иконки, цвета, шрифты — всё остаётся идентично.

### Добавляем: время прохождения теста

В таблице лидеров + на подиуме для каждого игрока отображается **время**, за которое он закончил тест — разница между `finish_time` и `start_time`. Это даёт быстрый ответ на вопрос "почему он 1-й, а я 2-й при одинаковом score" — потому что закончил на 8 секунд раньше.

**Формат отображения**:
- Если ≤ 60 секунд → `45 с` или `45 сек`
- Если > 60 секунд → `1:23` (минуты:секунды)
- Если игрок не закончил до истечения матча → показать символ `⏱ ―` или `не закончил`
- Если `has_left == true` → показать `⨯ вышел` (текущий бейдж, без времени)

**Где отображается**:
- В строке таблицы лидеров — новая колонка справа от score
- На подиум-карточке 1-го места — небольшой ярлык под именем ("за 3:12")
- В личном summary текущего игрока под ShieldPlaceAnimation — крупно: "Твоё время: 2:47"

### Контракт для клиента

Клиент вычисляет время так:

```dart
String formatTimeTaken(MemberDto member, DateTime? startTime) {
  if (member.hasLeft) return '⨯ вышел';
  final finish = member.finishTime;
  if (finish == null || startTime == null) return '―';
  final taken = finish.difference(startTime);
  if (taken.isNegative) return '―'; // invalid server data
  return taken.inSeconds <= 60
      ? '${taken.inSeconds} с'
      : '${taken.inMinutes}:${(taken.inSeconds % 60).toString().padLeft(2, '0')}';
}
```

Для корректной работы сервер **обязан** прислать `finish_time` (ISO-8601 UTC) для каждого доигравшего игрока (см. п. 2). Если игрок не доиграл (`has_left: true`) — `finish_time` может быть `null`, клиент отрендерит бейдж "⨯ вышел" вместо времени.

### Без чего НЕ работает

Отображение времени напрямую зависит от `finish_time` в payload-е `results`. Если поле `finish_time` не придёт — клиент просто скрывает колонку (или показывает `―` повсюду). Основная функциональность (ranking + призы + анимации) не пострадает.

## Чек-лист после внедрения

- [ ] В `start_game`: `end_time - start_time == 240 сек` (4 минуты)
- [ ] Сервер записывает `finish_time` при последнем `answered` или `finish_test` или истечении таймера
- [ ] Сортировка использует полную цепочку tie-breaker'ов из п. 3
- [ ] Поле `place` приходит в `MemberDto` в payload `results` (и `finish_test`, если используется)
- [ ] Поле `finish_time` приходит как ISO-8601 UTC для каждого доигравшего игрока (для отображения времени в UI)
- [ ] `won_coins` вычисляется по формуле 50/30/20 от `pool = money_count × count(real_human_members)` (боты в pool не входят)
- [ ] Округление — в пользу игрока (half-up) для 1-го и 2-го места; 3-е получает остаток. Инвариант: `first + second + third == pool`
- [ ] Боты (`id < 0`): `place = null`, `won_coins = 0`, в pool не входят, в нумерации мест не участвуют
- [ ] Ушедшие (`has_left: true`): `place = null`, `won_coins = 0`, но присутствуют в `results.members`
- [ ] Edge case: 3+ доигравших → 50/30/20; 2 → 70/30; 1 → 100%; 0 → pool сгорает
- [ ] Все 8 тест-кейсов выше проходят (включая округление на нечётном pool)
- [ ] Текущие анимации результатов (ShieldPlaceAnimation, подиум, таблица) сохранены — логика места берётся из поля `place`
- [ ] Время прохождения теста отображается в таблице и на подиуме (вычисляется клиентом из `finish_time - start_time`)

## Контакты

Клиентская сторона готова к приёму новых полей и проигнорирует их, если прилетит со старого бэкенда. Можно внедрять поэтапно — сначала `finish_time` + `place`, потом доработать формулу призов.
